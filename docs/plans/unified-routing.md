# Plan: Unified Annotation Routing — One Source of Truth

## Status: Partially implemented, then revised

### What was implemented
- Scanner emits `routes.generated.zig` from `// match` annotations (Step 1)
- Scanner emits `route_match` in manifest JSON
- TypeScript sidecar uses generated route table with `matchRoute()` from shared module
- Cross-language test vectors for route matching (22 cases)

### What was reverted and why
- `app.zig translate()` was changed to use the generated route table, then reverted
  to the original handler-iteration pattern. Reason: the generated table can't handle
  shared method+pattern routes (GET /products: list vs search). The Zig pipeline's
  `inline for (handlers)` with `route()` disambiguation handles this correctly.
- `search_products` URL was changed from `/products` to `/products/search`, then
  reverted. Reason: `/products?q=widget` is correct REST — filtering a collection
  by query param is the same endpoint, not a sub-resource.

### Revised understanding
The Zig handlers' `route_method`/`route_pattern` constants are the source of truth
for the Zig pipeline. The `// match` annotations are the source of truth for the
sidecar pipeline. The scanner validates both exist and can cross-check them. Full
unification (removing constants, using only annotations) requires solving the shared
pattern problem — either the generated table supports multiple handlers per pattern,
or the Zig pipeline gets a different dispatch mechanism. Deferred.

## Context

Today route metadata is declared twice:

**Zig handlers** — comptime constants:
```zig
pub const route_method = t.http.Method.get;
pub const route_pattern = "/products/:id";
```

**TypeScript handlers** — comment annotations:
```typescript
// [route] .get_product
// match GET /products/:id
```

The annotation scanner validates the TypeScript annotations. The Zig
compiler validates the Zig constants. Nobody validates they agree. A
handler can say `route_method = .post` while the annotation says
`// match GET /products`. Two checks verifying different data — a gap
in our pair assertion model.

## Decision

One source of truth: the `// match` annotation. The scanner extracts
routing metadata. Two generated files consume it:
- `generated/dispatch.generated.ts` — TypeScript route table (exists today)
- `generated/routes.generated.zig` — Zig route table (new)

Both languages use generated code from the same scanner output. No
hand-written route constants. No divergence possible.

## Why not `@embedFile` + comptime JSON

Zig can parse JSON at comptime via `@embedFile("manifest.json")` +
`std.json`. But comptime JSON parsing is slow and uses comptime memory.
24 handlers is fine. 200 is a comptime memory bomb. TigerBeetle moved
away from comptime-heavy patterns for data-driven things for this reason.

Generate a Zig file instead. A comptime-known array of structs has zero
parsing overhead, full type safety, and exhaustiveness checking by the
compiler.

## Implementation

### Step 1: Extend the annotation scanner to emit `routes.generated.zig`

The scanner already emits `manifest.json`. Add a second output:

```zig
// Auto-generated by annotation scanner — do not edit.
const message = @import("../message.zig");
const http = @import("http.zig");

pub const Route = struct {
    operation: message.Operation,
    method: http.Method,
    pattern: []const u8,
};

pub const routes = [_]Route{
    .{ .operation = .get_product, .method = .get, .pattern = "/products/:id" },
    .{ .operation = .create_product, .method = .post, .pattern = "/products" },
    .{ .operation = .search_products, .method = .get, .pattern = "/products/search" },
    // ... all 24 operations
};
```

**Scanner changes:**
- New `--routes-zig=<path>` flag
- Emit sorted by specificity (literal before param, longer before shorter — same sort as TypeScript adapter)
- Validate: every operation in the Zig handler directory has a `// match` annotation
- Validate: no duplicate method + pattern

### Step 2: Update `app.zig` translate() to use generated routes

Current:
```zig
inline for (handlers) |H| {
    const skip = method != H.route_method or
                 parse.match_route(path, H.route_pattern) == null;
    if (!skip) {
        if (H.route(method, path, body)) |msg| {
            result = msg;
        }
    }
}
```

New:
```zig
const gen = @import("generated/routes.generated.zig");

inline for (gen.routes) |route| {
    if (method == route.method) {
        if (parse.match_route(path, route.pattern)) |params| {
            // Look up handler by operation, call route()
            const handler = comptime handler_for_operation(route.operation);
            if (handler.route(method, path, body, params)) |msg| {
                result = msg;
            }
        }
    }
}
```

Same comptime inlining. Same zero-overhead pattern matching. Data
comes from the scanner, not from hand-written constants.

**Key: the handler's `route()` function is NOT removed.** The annotation
tells the framework *which* handler matches (method + pattern). The
handler's `route()` does *additional validation* (parsing JSON body,
checking field presence, returning null to decline). Both layers serve
different purposes.

### Step 3: Remove `route_method` / `route_pattern` from Zig handlers

Each handler file changes from:
```zig
pub const route_method = t.http.Method.get;
pub const route_pattern = "/products/:id";

pub fn route(method: http.Method, path: []const u8, body: []const u8) ?Message {
    const params = t.match_route(path, route_pattern) orelse return null;
    // ...
}
```

To:
```zig
// [route] .get_product
// match GET /products/:id

pub fn route(method: http.Method, path: []const u8, body: []const u8, params: RouteParams) ?Message {
    // params already extracted by framework — no match_route call needed
    // ...
}
```

The handler receives `params` pre-populated by the framework (matching
how TypeScript handlers receive `req.params`). No redundant match_route
call inside the handler.

### Step 4: Update Zig handler `route()` signature

Currently `route()` takes `(method, path, body)` and does its own matching.
New signature: `route(method, path, body, params)` — params pre-extracted.

This is a breaking change to all 24 Zig handlers. Each one:
- Remove `const params = t.match_route(...)` line
- Add `params` parameter
- Access params directly: `params.get("id")`

### Step 5: Update build.zig

Add dependency: scan step must run before compile step.

```zig
// Scan annotations → generate routes.generated.zig + manifest.json
const scan_step = b.addRunArtifact(scanner_exe);
scan_step.addArgs(&.{ "handlers/", "--manifest=generated/manifest.json",
    "--routes-zig=generated/routes.generated.zig" });

// Main executable depends on scan output
exe.step.dependOn(&scan_step.step);
```

### Step 6: Add `// match` annotations to all 24 Zig handlers

Each Zig handler already has the route info in constants. Move it to
annotations:

```zig
// Before:
pub const route_method = t.http.Method.get;
pub const route_pattern = "/products/:id";

// After:
// [route] .get_product
// match GET /products/:id
```

The scanner validates these at scan time. The generated
`routes.generated.zig` makes them available at compile time.

### Step 7: Comptime pair assertions

In `routes.generated.zig` or `app.zig`:

```zig
comptime {
    // Assert: every operation in the Operation enum has a route entry.
    for (std.enums.values(message.Operation)) |op| {
        var found = false;
        for (routes) |r| {
            if (r.operation == op) { found = true; break; }
        }
        assert(found); // missing // match annotation for operation
    }

    // Assert: no duplicate method + pattern.
    for (routes, 0..) |a, i| {
        for (routes[i+1..]) |b| {
            assert(a.method != b.method or
                   !std.mem.eql(u8, a.pattern, b.pattern));
        }
    }
}
```

Scanner validates at scan time. Compiler validates at compile time.
Two independent checks on the same data — pair assertions.

## Verification

1. `zig build scan` generates both `manifest.json` and `routes.generated.zig`
2. `zig build run` compiles with generated routes — all comptime assertions pass
3. `zig build test` — sim tests pass (routing works end-to-end)
4. `zig build fuzz -- smoke` — fuzz tests pass
5. TypeScript sidecar still works (manifest unchanged, dispatch unchanged)
6. Cross-language: same `// match` annotation in both Zig and TypeScript handlers

## What breaks

- All 24 Zig handlers: remove constants, add `// match` annotation, update `route()` signature
- `app.zig`: translate() rewritten to use generated routes
- `build.zig`: scan step becomes a dependency of compile step
- `prelude.zig`: remove `match_route` re-export (handlers no longer call it directly)
- Handler tests that call `route()` directly: update to pass params

## What doesn't break

- TypeScript handlers: unchanged (already use annotations)
- Annotation scanner: extended, not rewritten
- Dispatch generator: unchanged (reads same manifest)
- Binary protocol: unchanged
- All framework code below routing: unchanged
