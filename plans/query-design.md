# Query Design — User-Defined Prefetch Queries

## Problem

The current sidecar interface only supports entity-level operations: get by ID, list, create, update, delete. The user can't express business queries like "low stock products," "revenue this month," or "orders pending longer than 24 hours." The native Zig path can (hand-written prefetch functions in state_machine.zig), but the sidecar path can't. This breaks the 1:1 parity goal.

## Design: SQL in prefetch, structs in commit

The prefetch/execute split already enforces the safety boundary. Data is frozen before the handler runs. Writes are atomic per tick. Single-threaded, no races. The architecture provides the guarantees — the query language doesn't need to.

- **Prefetch** — user writes SQL. Full flexibility: filters, joins, aggregations, date math. The framework runs it, results land in typed cache slots.
- **Commit** — user returns struct writes only. `{ update_product: { id, inventory } }`. The framework generates the mutation SQL.

```ts
// [prefetch] .low_stock_report
sql: "SELECT * FROM products WHERE active = 1 AND inventory < 10 ORDER BY inventory LIMIT 50"

// [handle] .low_stock_report
function handleLowStockReport(ctx) {
  return { status: "ok", writes: [] };  // read-only
}

// [render] .low_stock_report
function renderLowStockReport(status, ctx) {
  return ctx.products.map(p => `<div>${p.name}: ${p.inventory} left</div>`).join("");
}
```

Build step: run the query against SQLite via temp view + `PRAGMA table_xinfo` to validate columns and generate a typed result struct. No SQL parser needed — SQLite does the parsing.

## Why this is safe

A declarative query object (`{ where: { inventory: { lt: 10 } } }`) was considered as a constrained alternative to SQL. But the constraint doesn't add safety — the architecture already provides it:

- Data is frozen before the handler runs (prefetch/execute split)
- Handler can't interleave reads and writes
- Writes are atomic per tick
- Single-threaded, no races
- Mutations go through typed structs, not SQL

Whether the user writes a struct filter or a SQL string in prefetch, the guarantees are identical. The struct was solving a problem the architecture already solved.

## Why mutations are struct-only

The dangerous bugs in web apps live in mutations. The framework eliminates them by owning the mutation SQL.

Traditional web app:

```sql
BEGIN;
  SELECT inventory FROM products WHERE id = :source FOR UPDATE;
  SELECT inventory FROM products WHERE id = :target FOR UPDATE;
  UPDATE products SET inventory = inventory - :qty WHERE id = :source;
  UPDATE products SET inventory = inventory + :qty WHERE id = :target;
COMMIT;
```

Bugs: race conditions between SELECT and UPDATE, forgot FOR UPDATE, partial commit on crash, negative inventory from stale reads.

Tiger — same operation:

```ts
function handleTransferInventory(ctx, body) {
  if (ctx.source.inventory < body.quantity) return { status: "insufficient_inventory", writes: [] };
  return {
    status: "ok",
    writes: [
      { update_product: { id: ctx.source.id, inventory: ctx.source.inventory - body.quantity } },
      { update_product: { id: ctx.target.id, inventory: ctx.target.inventory + body.quantity } },
    ]
  };
}
```

Every bug is eliminated by architecture: single-threaded (no races), prefetch freezes the snapshot (no stale reads), writes apply atomically (no partial commits). The SQL the framework generates is trivial — `UPDATE products SET inventory = ? WHERE id = ?`.

## Database portability

- **Struct writes** — framework generates the SQL. Swap databases, user code doesn't change.
- **Prefetch SQL** — user owns the SQL string. Swap databases, user rewrites their queries.

The framework does not provide a SQL transpiler or dialect abstraction. That's deliberate — dialect layers are massive surface areas that every ORM struggles with. The user who writes SQL in prefetch already knows their dialect. Switching databases means rewriting a handful of read queries, not the entire app.

## Result type generation via database adapter

The result struct for custom SQL queries is generated by asking the database, not by parsing SQL. The adapter is thin plumbing — the database is the source of truth:

```
User writes:  "SELECT name, SUM(quantity) as total_sold FROM ..."
Build step:   CREATE TEMP VIEW AS <user sql>; PRAGMA table_xinfo(temp_view);
SQLite says:  name TEXT, total_sold INTEGER
Codegen:      struct { name: [256]u8, name_len: u8, total_sold: u32 }
```

SQLite parses the SQL, resolves the types, and the build step reads the answer. The adapter has no bug surface — it's not inferring types, it's relaying what the database already knows.

For other databases, the adapter calls the equivalent introspection:
- Postgres: `pg_catalog` views
- MySQL: `DESCRIBE`

Each adapter is ~20 lines of "ask the database what the columns are." This follows the sidecar adapter pattern — each language has an adapter (`adapters/typescript.ts`), each database has an adapter. Thin glue between the framework and the target.

## Testing trade-off

SQL prefetch queries can't run against `MemoryStorage` (flat arrays, no SQL engine). This means no equivalence testing for custom queries. The auditor can't independently validate them.

This is acceptable because:
- Custom SQL queries are reads. Wrong data on a dashboard is a bug report, not data corruption.
- Mutations go through typed structs with full auditor coverage. That's where the money moves.
- The user can provide a hand-written mock for auditor validation (opt-in, not required).

TigerBeetle needs equivalence testing on every operation because a wrong financial balance compounds silently. A wrong product count on a web page gets corrected on the next refresh.

## Routes considered and rejected

**Declarative query object** (`{ where: { inventory: { lt: 10 } } }`): Enables auto-generated MemoryStorage filters for equivalence testing. Rejected — the architecture already provides the safety guarantees, and the declarative language is a new surface area to design and maintain for marginal benefit on simple queries.

**Expose storage object to handlers**: Breaks the prefetch/execute split. The handler is no longer a pure function. Rejected.

**ORM integration**: The user never touches storage — that's the framework's value proposition. Adding an ORM contradicts the architecture. Rejected.

**Drop MemoryStorage, use SQLite :memory: for all tests**: Simpler but loses PRNG-driven fault injection for busy/err paths. Deferred.

**AI-generated mocks from SQL**: The AI produces the same logic in two syntaxes, not two independent implementations. Same bug in both. Doesn't provide the equivalence testing benefit. Rejected.

## Convention: assert the prefetch contract at handler entry

The handler should assert what the query promised before using the data. This is the same pattern as Zig-side assertions — validate at the boundary, trust inside. The prefetch SQL is an external input to the handler. Assert it.

```ts
// [handle] .low_stock_report
function handleLowStockReport(ctx) {
  for (const p of ctx.products) {
    assert(p.active);
    assert(p.inventory < 10);
  }
  return { status: "ok", writes: [] };
}

// [handle] .transfer_inventory
function handleTransferInventory(ctx, body) {
  assert(ctx.source !== null);
  assert(ctx.target !== null);
  assert(ctx.source.id !== ctx.target.id);

  if (ctx.source.inventory < body.quantity) return { status: "insufficient_inventory", writes: [] };
  return {
    status: "ok",
    writes: [
      { update_product: { id: ctx.source.id, inventory: ctx.source.inventory - body.quantity } },
      { update_product: { id: ctx.target.id, inventory: ctx.target.inventory + body.quantity } },
    ]
  };
}
```

This catches wrong rows, missing filters, soft-delete leaks, and type mismatches — the common query bugs. No auditor, no mocks, no second implementation. One line per assumption.

The bigger value: writing the assertion forces the user to state what they expect. "I expect all these products to be active" — if they can't write that assertion, they don't understand their own query. The assert is a thinking tool, not just a runtime check.

### What assertions don't catch: sequence-dependent bugs

Assertions verify each operation in isolation. The auditor verifies cumulative state across thousands of operations. Example: a transfer handler with a rounding bug that loses 1 inventory unit every 100th transfer. Each operation's assertions pass — inventories look reasonable, both non-negative. But after 1,000 transfers, 10 units have vanished. No single assertion catches it because no single operation is wrong.

The auditor catches it because it tracks running totals independently. After the 100th transfer, its total diverges from the state machine's. Assert fires with the exact seed and operation.

### What can be generated vs what can't

**CRUD auditor (generated from entity declarations):** The auditor for create, get, update, delete, list is mechanical — store on create, check on get, update on update, remove on delete, compare on list. The framework generates this from the entity types. Catches missing rows, phantom rows, wrong field values after sequences of CRUD operations, soft-delete not filtering from lists, version not incrementing. Fully automatic, no user effort.

**Business logic auditor (hand-written or AI-generated):** The transfer inventory example requires knowing that the handler subtracts quantity from source and adds to target. The framework can't generate this from the struct — the struct doesn't encode the computation. This is where the cloud-generated AI auditor or a hand-written mock fills the gap.

The generated CRUD auditor covers the majority of operations and catches real bugs without the user doing anything. The business logic auditor is opt-in for the few custom operations where cumulative correctness matters.

## Conclusion: queries are domain logic

SQL in prefetch is domain logic — "what does low stock mean" is a business decision, not a framework concern. The framework's boundary is clear:

**Framework guarantees (mechanical):**
- Whatever the query returns, it's frozen before the handler runs
- Writes are atomic per tick
- The pipeline is correct (connections, HTTP, SSE, WAL, auth, ordering)
- Struct writes can't produce dangerous SQL

**User's responsibility (domain):**
- The query asks the right question
- The handler makes the right decision
- The render shows the right thing

The framework makes it easy to write correct queries (typed results, build-time schema validation) and impossible to corrupt state (struct writes only). It does not verify that `inventory < 10` is the right threshold. That's the user's domain, not the framework's job.

Trying to verify the user's SQL is the framework trying to do the user's job — that's the complexity trap every ORM falls into. The auditor exists as an opt-in testing tool for users who want extra confidence (and as a cloud-generated feature for those who don't want to maintain it), not as a framework guarantee.

## Is this as safe as TigerBeetle?

For our domain, yes. The safety comes from different layers for different concerns:

- **Pipeline correctness** — prefetch/execute split, atomic commits, single-threaded. Same as TB. Mechanical guarantee.
- **CRUD correctness** — generated auditor from entity declarations. Same coverage as TB for entity operations. Automatic.
- **Query correctness** — assertions at handler entry. TB doesn't have user-defined queries, so they don't have this problem.
- **Mutation correctness** — struct writes only. Framework generates the SQL. Same guarantee as TB.

TB's extra guarantee (equivalence testing across storage backends) tests their LSM tree implementation. We trust SQLite — we didn't write it. They can't trust theirs without testing it — they did write it.

The TigerBeetle team wouldn't argue theory. They'd say: "Show us the seed that breaks it." Run the fuzzer — thousands of seeds, random operation sequences, auditor checking every response. If no seed breaks it, the design holds. If one does, fix the bug and add the seed to CI.
