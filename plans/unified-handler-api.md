# Unified Handler API

Single user space spec for declaring operations. Annotations are the user-facing contract — same format, same scanner, same mental model for Zig and TypeScript. The scanner is the authority for discovery and exhaustiveness. But the generated Zig code is structured so the compiler does maximum validation after generation: type checking, signature matching, prefetch→handle→render data flow.

Two layers of validation, each doing what it's best at:
- **Scanner catches**: missing operations, duplicate phases, invalid operation names, missing function bodies on mutations
- **Compiler catches**: wrong types, wrong signatures, broken data flow between phases, prefetch/handle/render type mismatches

---

## 1. User space spec

What the user writes. Language-agnostic, same for Zig and TypeScript.

### Annotation format

```
// [route] .operation_name
// [prefetch] .operation_name
// [handle] .operation_name       (function follows = mutation, no function = read-only)
// [render] .operation_name
```

### Scanner rules

- Annotation must be followed by non-empty, non-comment code line (except `[handle]` — function body is optional)
- Operations must be valid enum values from `message.Operation`
- Every non-root operation must have all 4 phases (route, prefetch, handle, render)
- No duplicate operation/phase pairs
- `[handle]` with function body → manifest marks as mutation
- `[handle]` without function body → manifest marks as read-only, framework generates passthrough
- `[handle]` without function body must be immediately followed by the next annotation or EOF — no stray code can appear between a skipped handle and the next phase
- `[prefetch]` uses `db.sql` as the primitive in both Zig and TypeScript. Zig calls storage directly in-process. TS calls storage over the unix socket via the SDK. Same SQL, same contract.
- `[prefetch]` function body is optional in TS (no function = framework handles it natively for simple cases)
- No required file structure — annotations are self-identifying, scanner finds them in any `.zig` or `.ts` file. Defaults to project root, `--path` flag overrides for subdirectory scoping.

### Handler context

Handle and render receive one `ctx` struct containing everything the handler needs:
- `ctx.prefetched` — domain data from prefetch (per-handler named struct)
- `ctx.body` — typed request body
- `ctx.identity` — framework-provided visitor fingerprint (`userId`, `isNew`, `isAuthenticated`)

Identity is not auth — the framework fingerprints the visitor, the user builds their own auth/permissions on top. Context types are comptime-generated by the framework from the handler's Prefetch type + Operation.EventType + Identity.

### Handler examples — Zig

#### Read-only operation (get_product)

```zig
pub const GetProductPrefetch = struct {
    product: ?Product,
};

// [route] .get_product
pub fn routeGetProduct(method: http.Method, path: []const u8, body: []const u8) ?message.Message {
    if (method != .get) return null;
    const id = matchPath("/products/{uuid}", path) orelse return null;
    return message.Message.init(.get_product, id, {});
}

// [prefetch] .get_product
pub fn prefetchGetProduct(db: *DB, msg: *const message.Message) ?GetProductPrefetch {
    const product = db.sql("SELECT * FROM products WHERE id = ?", .{msg.body_as(Product).id});
    return .{ .product = product };
}

// [handle] .get_product

// [render] .get_product
pub fn renderGetProduct(ctx: GetProductContext, buf: []u8) usize {
    // ctx.prefetched, ctx.identity available
    // write HTML into buf, return bytes written
}
```

#### Mutation (create_product)

```zig
pub const CreateProductPrefetch = struct {
    existing: ?Product,
};

// [route] .create_product
pub fn routeCreateProduct(method: http.Method, path: []const u8, body: []const u8) ?message.Message {
    if (method != .post) return null;
    if (!mem.eql(u8, path, "/products")) return null;
    return codec.parseProduct(body);
}

// [prefetch] .create_product
pub fn prefetchCreateProduct(db: *DB, msg: *const message.Message) ?CreateProductPrefetch {
    const existing = db.sql("SELECT * FROM products WHERE id = ?", .{msg.body_as(Product).id});
    return .{ .existing = existing };
}

// [handle] .create_product
pub fn handleCreateProduct(ctx: CreateProductContext) ExecuteResult {
    if (ctx.prefetched.existing != null) return ExecuteResult.read_only(.version_conflict);
    return ExecuteResult.single(.ok, .{ .put_product = ctx.body.* });
}

// [render] .create_product
pub fn renderCreateProduct(ctx: CreateProductContext, buf: []u8) usize {
    // write HTML into buf
}
```

#### Complex mutation (create_order)

```zig
pub const CreateOrderPrefetch = struct {
    products: [20]?Product,
    inventory: [20]u32,
};

// [route] .create_order
pub fn routeCreateOrder(method: http.Method, path: []const u8, body: []const u8) ?message.Message {
    if (method != .post) return null;
    if (!mem.eql(u8, path, "/orders")) return null;
    return codec.parseOrder(body);
}

// [prefetch] .create_order
pub fn prefetchCreateOrder(db: *DB, msg: *const message.Message) ?CreateOrderPrefetch {
    const order = msg.body_as(Order);
    const products = db.sql("SELECT * FROM products WHERE id IN (?)", .{order.item_ids()});
    const inventory = db.sql("SELECT product_id, count FROM inventory WHERE product_id IN (?)", .{order.item_ids()});
    return .{ .products = products, .inventory = inventory };
}

// [handle] .create_order
pub fn handleCreateOrder(ctx: CreateOrderContext) ExecuteResult {
    // check inventory, build writes array (up to 21: 1 order + 20 inventory updates)
}

// [render] .create_order
pub fn renderCreateOrder(ctx: CreateOrderContext, buf: []u8) usize {
    // write HTML into buf
}
```

### Handler examples — TypeScript

```typescript
// [route] .create_product
export function routeCreateProduct(req: Request): Route<Product> | null { ... }

// [prefetch] .create_product
export async function prefetchCreateProduct(db: DB, body: Product): Promise<CreateProductPrefetch> {
    const existing = await db.sql("SELECT * FROM products WHERE id = ?", [body.id]);
    return { existing };
}

// [handle] .create_product
export function handleCreateProduct(ctx: CreateProductContext): HandlerResult {
    ctx.prefetched  // domain data from prefetch
    ctx.body        // typed Product from request
    ctx.identity    // { userId, isNew, isAuthenticated }
}

// [render] .create_product
export function renderCreateProduct(ctx: CreateProductContext): string { ... }
```

4 annotations in both languages. Same `db.sql` primitive, same `ctx` shape. Zig calls storage in-process, TS calls storage over the unix socket. Same SQL, same contract.

### Render modes

Renders have two modes, determined by the function signature:

**String return** — produces HTML. Used for page loads and simple mutations.

```typescript
// [render] .page_load_dashboard
export function renderDashboard(ctx: DashboardContext): string {
    return `<html>...</html>`;
}
```

```zig
// [render] .page_load_dashboard
pub fn renderDashboard(ctx: DashboardContext, buf: []u8) usize {
    // write HTML into buf, return bytes written
}
```

**RenderEffects return** — composes UI effects as data. Used when a mutation needs to update multiple parts of the page, push notifications, or sync other clients. The render returns a list of effect tuples; the framework executes them after the response is sent.

```typescript
// [render] .create_product
export function renderCreateProduct(ctx: CreateProductContext): RenderEffects {
    return render([
        ["patch", "#toast", '<div class="toast">Product created</div>', "append"],
        ["sync", "/dashboard"],
    ]);
}
```

```zig
// [render] .create_product
pub fn renderCreateProduct(ctx: CreateProductContext) RenderEffects {
    return render(.{
        .{ "patch", "#toast", "<div class=\"toast\">Product created</div>", "append" },
        .{ "sync", "/dashboard" },
    });
}
```

Tuple syntax — readable in diffs, AI-friendly, compiler catches type mismatches. Each tuple is an independent effect. The framework inspects the list, can deduplicate (two `sync("/dashboard")` becomes one), batch, and execute in order after the response is sent.

Effect types align 1:1 with Datastar SSE events, generated from a comptime spec:

| Tuple prefix | Datastar event | Purpose |
|---|---|---|
| `"patch"` | `datastar-patch-elements` | Morph/append/replace/remove DOM elements |
| `"signal"` | `datastar-patch-signals` | Update reactive state on client |
| `"script"` | `datastar-patch-elements` | Execute JS in the browser |
| `"remove"` | `datastar-patch-elements` | Remove elements by selector (sugar for patch + remove mode) |
| `"sync"` | *(framework-level)* | Re-run a page's prefetch → render, push to all SSE subscribers |

Patch modes: `outer` (default), `inner`, `replace`, `prepend`, `append`, `before`, `after`, `remove`.

For the rare cases needing optional params (view transitions, onlyIfMissing), an options object goes at the end of the tuple. 95%+ of usage needs only the positional args.

Start with string returns. When a mutation needs effects, return `RenderEffects` instead. No annotation changes — just a different return type. See [live partial updates recipe](../docs/recipes/live-partial-updates.md).

## 2. Build pipeline

How annotations become executable code.

### Pipeline overview

```
handlers/products.zig              handlers/products.ts
  // [route] .get_product            // [route] .get_product
  // [prefetch] .get_product
  // [handle] .get_product           // [handle] .get_product
  // [render] .get_product           // [render] .get_product
         |                                  |
         +----------------+-----------------+
                          |
                   annotation scanner
                   (zig build scan)
                          |
                   manifest.json
                          |
              +-----------+-----------+
              |                       |
        Zig adapter              TS adapter
              |                       |
   handlers.generated.zig    dispatch.generated.ts
              |                       |
    comptime validation        unix socket server
    direct function calls      serialized protocol
    inlined by compiler        separate process
```

The manifest is the universal interchange format: single test surface (test the scanner once, test each adapter against the manifest format), debuggable (readable JSON), and each piece is simple and independently testable.

### Generated Zig dispatch (handlers.generated.zig)

```zig
// AUTO-GENERATED by zig adapter — do not edit
const products = @import("../handlers/products.zig");
const orders = @import("../handlers/orders.zig");

pub const handlers = .{
    .{
        .operation = .get_product,
        .route = products.routeGetProduct,
        .prefetch = products.prefetchGetProduct,
        .Prefetch = products.GetProductPrefetch,
        .execute = .read_only,  // no function body after [handle]
        .render = products.renderGetProduct,
    },
    .{
        .operation = .create_product,
        .route = products.routeCreateProduct,
        .prefetch = products.prefetchCreateProduct,
        .Prefetch = products.CreateProductPrefetch,
        .execute = products.handleCreateProduct,
        .render = products.renderCreateProduct,
    },
    .{
        .operation = .create_order,
        .route = orders.routeCreateOrder,
        .prefetch = orders.prefetchCreateOrder,
        .Prefetch = orders.CreateOrderPrefetch,
        .execute = orders.handleCreateOrder,
        .render = orders.renderCreateOrder,
    },
};

// Context types are comptime-generated by the framework from Prefetch + Operation.EventType + Identity
// e.g., CreateProductContext = struct { prefetched: CreateProductPrefetch, body: *const Product, identity: Identity }
```

### Build commands

```bash
zig build scan          # scanner + adapter — generates manifest + handlers.generated.zig
zig build run           # compile + run (no scanning — uses committed generated files)
zig build test          # compile + run tests
```

Generated files (`manifest.json`, `handlers.generated.zig`) are committed to the repo. `zig build run` just compiles — no scanning, no generation. The user runs `zig build scan` after changing annotations. CI runs `zig build scan` to verify generated files aren't stale.

The scanner defaults to project root. Override with `--path` for subdirectory scoping:

```bash
zig build scan -- --path=examples/ecommerce-ts/handlers/
```

## 3. Framework internals

How the framework consumes the generated handler tuple.

### Framework dispatch (comptime-derived)

Dispatch functions are generated by the framework's `App()` comptime function as methods on the returned type. The framework iterates the handler tuple for validation and generates dispatch in the same pass. Matches TigerBeetle's pattern: comptime functions return types that include all derived behavior.

```zig
// app.zig
pub const App = framework.App(.{
    .handlers = @import("generated/handlers.generated.zig").handlers,
    .Message = message.Message,
    .MessageResponse = message.MessageResponse,
});

// App now has: App.translate, App.dispatch_prefetch, App.dispatch_execute, App.dispatch_render
// All comptime-generated from the handler tuple. No separate dispatch file.
```

The framework builds these at comptime:

```zig
fn translate(method: Method, path: []const u8, body: []const u8) ?Message {
    // In test/debug builds: try every route, assert at most one matches (catches duplicate routes).
    // In release: early return on first match (route table is static, fully exercised by fuzz tests).
    var result: ?Message = null;
    inline for (handlers) |H| {
        if (H.route(method, path, body)) |msg| {
            if (is_release) return msg;
            assert(result == null); // duplicate route match
            result = msg;
        }
    }
    return result;
}

fn dispatch_prefetch(comptime op: Operation, storage: *Storage, msg: *const Message) ?PrefetchUnion {
    const H = handler_for(op);
    const result = H.prefetch(storage, msg) orelse return null; // busy, retry
    return @unionInit(PrefetchUnion, @tagName(op), result);
}

fn dispatch_execute(comptime op: Operation, prefetched: PrefetchUnion, msg: *const Message, identity: Identity) ExecuteResult {
    const H = handler_for(op);
    const data = @field(prefetched, @tagName(op));
    const ctx = H.Context{ .prefetched = data, .body = msg.body_as(op.EventType()), .identity = identity };
    if (H.execute == .read_only) return ExecuteResult.read_only(ctx.prefetched);
    return H.execute(ctx);
}
```

`PrefetchUnion` is a comptime-generated tagged union of all handlers' prefetch types. Stored on the connection between ticks.

After migration, `codec.zig`, `state_machine.zig` dispatch switches, and `render.zig` dispatch switches are deleted. The framework's comptime-derived `App` type replaces them entirely.

### Comptime validation

The framework imports `handlers.generated.zig` and at comptime validates structural correctness, type flow, and semantic constraints. The scanner catches missing/duplicate operations; the compiler catches everything else.

#### Structural

1. **Exhaustiveness** — every `Operation` (except `.root`) has exactly one handler entry
2. **No duplicates** — no two entries claim the same operation
3. **Handler count** — `handlers.len == @typeInfo(Operation).@"enum".fields.len - 1` (one-line invariant, catches drift immediately)
4. **Signature check** — route/prefetch/execute/render functions have correct parameter and return types

#### Type flow (prefetch → handle → render)

5. **Prefetch→handle→render type chain** — assert return type of prefetch matches `ctx.prefetched` in handle and render. Not convention — a compile error if they diverge.
6. **EventType↔prefetch body access** — assert that when prefetch calls `msg.body_as(T)`, `T` matches `Operation.EventType` for that operation. Catches prefetch reading the message body as the wrong type.
7. **Operation↔route agreement** — generated wrapper asserts the returned message's `.operation` field matches the declared operation. Catches a route that accidentally returns the wrong operation tag.

#### Write safety

8. **Read-only passthrough** — if scanner marked no handle function, the generated passthrough is `ExecuteResult{ .writes_len = 0 }` at comptime. The compiler proves no writes exist rather than trusting runtime.
9. **ExecuteResult write type constraints** — constrain which write variants are valid for a given operation. `create_product` can only produce `.put_product` writes, not `.put_order`. Currently any handler can return any write type — generated code makes this a compile error.
10. **Cross-operation write awareness** — if two handlers produce writes to the same entity type (e.g., both `create_order` and `update_inventory` write `.put_inventory`), comptime flags this for explicit acknowledgment. Not an error — forces the developer to consider concurrent write semantics.

#### Prefetch contract

11. **Prefetch struct completeness** — every field in the Prefetch struct must be populated in all return paths. The compiler proves this naturally when the struct is constructed with `.{ .field = value }` syntax — missing fields are a compile error.
12. **Prefetch null = storage busy only** — `null` from prefetch is the retry signal (storage busy). Domain-level "not found" must go through the struct's `result` field. The generated wrapper can enforce this by typing the return as `?H.Prefetch` where the struct always contains a `result` field.

Failures are compile errors with source-mapped messages pointing at the handler file, not the generated file.

#### Staleness is self-correcting

A stale generated file always fails to compile. If the scanner didn't re-run after a change:
- Renamed function → generated `@import` references a symbol that doesn't exist → compile error
- Changed type → generated code passes wrong type to comptime check → compile error
- New operation added to enum → exhaustiveness check finds no handler entry → compile error
- Removed operation → generated code references non-existent enum variant → compile error

The compiler is the assertion. Even if build step ordering is wrong, the compiler rejects stale output. No extra staleness-detection mechanism needed.

## 4. Testing boundary

The framework owns pipeline correctness — fuzz tests with PRNG-driven fault injection, auditor oracle, full-stack simulation. This covers routing, dispatch, storage, WAL, connection lifecycle.

The user owns domain correctness. Per-handler functions are pure enough to test directly — handle takes a ctx struct, returns an ExecuteResult. No I/O, no framework setup. The framework provides the testable shape; the user writes the meaningful assertions for their business logic in their language of choice (Zig's built-in test, vitest, jest, etc.).

## 5. Migration plan

### Phase 1: Design validation (get_product + create_order)
Port the simplest read (`get_product`) and hardest mutation (`create_order`) to the new format. Build the full pipeline: scanner recognizes Zig annotations, Zig adapter generates dispatch, framework validates at comptime, handlers execute correctly. These two operations exercise every edge: skipped handle, simple prefetch, multi-entity prefetch, write arrays.

### Phase 2: Zig adapter
Build `adapters/zig.zig` (or extend annotation_scanner.zig) to emit `handlers.generated.zig` from the manifest. Wire into `build.zig` so scanner + adapter run before compilation.

### Phase 3: Framework comptime validation
Add the validation layer in the framework that imports the generated file, checks exhaustiveness, signatures, type flow. Source-map errors back to handler files.

### Phase 4: Port remaining operations
Move operations one at a time from the current switch statements to handler files. Old switches shrink as handlers are ported. Fallback: unported operations still work through the existing code path.

### Phase 5: Remove old code paths
Once all operations are ported, delete codec.zig dispatch switches, state_machine.zig inline switches, render.zig encode switches. The framework dispatch replaces them entirely.

## 6. What this replaces

Current code has `FollowupState`, `encode_followup()`, `refresh_message()`, and `needs_followup()` — all explicit per-operation SSE wiring. The `RenderEffects` struct replaces all of it. Render returns data describing effects, the framework executes them. Pure function, testable, serializable over the sidecar protocol.

## Design decisions

1. **Annotations are the single user space spec.** Both Zig and TypeScript handlers use the same `// [phase] .operation` format. The annotation scanner is the source of truth for what operations exist and what phases are implemented. Language doesn't matter — the spec is the same.

2. **Manifest is the universal interchange format.** The scanner always emits JSON, per-language adapters consume it. Single test surface, debuggable, each piece independently testable — even though the Zig adapter could be collapsed into the scanner.

3. **Generated code maximizes compiler power.** Annotations handle discovery; the compiler handles correctness. The Zig adapter generates `@import`-based dispatch that gives the compiler full visibility.

4. **Per-handler prefetch return types.** The return type IS the contract — the compiler enforces the link. No shared flat cache visible to user space.

5. **Single `ctx` parameter.** Framework assembles `ctx` from pieces it already has. Identity is not auth — the framework fingerprints the visitor.

6. **Handle annotation required, function optional.** The code is the truth — no modifiers, no separate `is_mutation()` classification.

7. **No required file structure.** Scanner finds annotations wherever they are.

8. **Dispatch is comptime-derived.** `framework.App()` returns a type with all dispatch methods. No separate dispatch file. Matches TigerBeetle's pattern.

9. **Render return type determines mode.** String return for static HTML, `RenderEffects` tuple list for composable effects. Both are pure functions returning data. No annotation to distinguish them. Effect types generated from a comptime spec aligned 1:1 with Datastar SSE events.
